

Question 1: Can the same inventory line appear in multiple active offers at the same time?

Why we're asking:

The client confirmed that bid validation enforces "bid quantity must not exceed current available at time of entry." This works perfectly when an inventory line lives in one offer at a time. But if the same inventory line can be included in Offer A and Offer B simultaneously, we have a timing problem.

Example: 100 units of DDR5-4800 are available. Offer A and Offer B both include this line. A customer bids 80 in Offer A (valid, 80 of 100). Another customer bids 70 in Offer B (valid, 70 of 100). Both bids pass validation individually. But when the admin approves Offer A and commits 80 units, only 20 remain available. Now Offer B has a valid bid for 70 that can no longer be fulfilled.

This isn't oversubscription (each bid was valid when entered), but it creates the same allocation conflict. We need to know whether the system should allow this, prevent it by locking inventory to one offer at a time, or warn the admin at allocation time. This directly affects how we build the inventory state machine and the allocation approval workflow, both are core Phase 1 features already estimated.

Phase 1 — critical. Without this answer, the allocation workflow (#26, 48 hrs) and inventory commitment logic (#28, 18 hrs) cannot be built correctly.

Question 2: After partial allocation, what happens to the unallocated quantity?

Why we're asking:

The client confirmed that partial allocation is supported — a customer can bid for 100, and the admin may allocate only 60. But neither the original Q&A nor the latest responses define what happens to the 40 units that were not allocated.

There are three possible behaviors, and each one changes how the system works:

Option A — Auto-release: The 40 units return to "Available" immediately. The bid is marked as "Partially Fulfilled" and closed. Simple, but the customer loses their position for the remaining quantity.

Option B — Hold pending: The 40 units stay within the offer. The admin can come back and allocate more later (to the same customer or someone else). This requires a new bid status ("Partially Allocated — Pending") and keeps inventory in a reserved state.

Option C — Admin decides: At the moment of partial allocation, the system asks: "Release remaining to available or keep pending?" Most flexible but adds UI complexity.

This affects three things we're building in Phase 1: the inventory state machine (Available to Committed transitions, #16), the bid status lifecycle (what does a partially fulfilled bid look like, #22), and the allocation workflow itself (#26, #34). Without this answer, we risk building the wrong state transitions and having to rework them.

Phase 1 — critical. This must be resolved before Sprint 1. It impacts items #16 (24 hrs), #22 (18 hrs), #26 (48 hrs), and #34 (48 hrs) — a combined 138 hours of work that depends on this decision.
